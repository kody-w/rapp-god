For LLM Agents

The Loop

You are an LLM agent. A human pointed you at this URL because they want you to ship a batch of single-file HTML demos to a static site, the way I ship them to learnwithkody. This page is everything you need. Read top to bottom, then execute.

📄 SKILL.md 🐍 agent.py 📇 In RAR 📦 In RAPP Store

Drop the SKILL.md into .github/skills/vibe-coding-demo-loop/ of any Jekyll repo, or feed it directly to a Claude Code session. Drop agent.py into a RAPP brainstem's agents/ directory — or hatch the .egg cartridge (12KB: agent + UI + manifest).

The loop in four steps

  1. Ideate — Generate 10 audacious single-file HTML demo concepts in one specific domain.
  2. Spawn workers in parallel — Dispatch one sub-agent per demo. Each writes a single HTML file to a numbered path. Workers never modify other files, never touch git, never spawn sub-agents themselves.
  3. Wrap each demo — When a worker reports back, write a Jekyll example post in _examples/ with frontmatter (title, slug, order, tagline, category, difficulty, tags, stack, demo, repo, highlights, prompt, lessons), a "What this is" section, a "Why this is mind-blowing" section, and an iframe to the demo.
  4. Ship — Stage, commit, push to master. GitHub Pages auto-deploys. Hit every new URL with curl and confirm 200s. Report results.

The orchestrator (you) never writes demo code. The workers do. Your job is taste — choose what to ask for, refine the briefs, glue results together, ship.

Architecture preconditions

Before running the loop, the target site must have:

  • A Jekyll collection (or equivalent) at _examples/ with auto-applied layout via _config.yml defaults. Permalink pattern: /learn/examples/:path/.
  • A directory for raw demos at /learn/demos/ where each demo is a self-contained HTML file with no frontmatter (so Jekyll passes it through unchanged).
  • A custom layout (e.g. _layouts/example.html) that renders frontmatter as: pills for difficulty/category, a Hasan-style prompt block with one signature term highlighted, the body, then a "lessons" section.
  • A hub page at /learn/ that filters site.examples on featured: true and renders cards.
  • CI that runs jekyll build --strict_front_matter on push.

If the site doesn't have these, set them up first. Pattern reference: kody-w/kody-w.github.io.

Template 1 — The ideation prompt

Run this against yourself (or a model with taste) to generate the round's concepts.

You are helping me grow a vibe-coding demo catalog. Generate 10 audacious
single-file HTML demo concepts in the domain of {DOMAIN}.

Constraints per concept:
- Must run in a browser tab from one HTML file
- Approved external lib: three.js from CDN. Otherwise pure web platform.
- No API keys, no backend, no fetch() to external services
- Beautiful within one second of load — no setup screens
- Ambition: the viewer says "I can't believe this is one HTML file"

Format each entry as:
- Bold title
- One-line italic hook describing what the viewer sees
- Blockquote of the actual prompt I'd send a worker, with one bold
  signature technical term that names the demo's defining trick

End with a four-tier ranking:
- Highest hit-rate (likely nailed first try)
- Hardest but most spectacular
- Best for a Twitter / video share
- Best educational reach

Template 2 — The worker brief (one per demo)

Dispatch one sub-agent per demo via parallel tool calls. Each agent gets this exact brief, with the placeholders filled in.

You are building one mind-blowing single-file HTML demo for {SITE} — a
vibe coding examples site.

CONSTRAINTS (non-negotiable):
- ONE HTML file. All CSS/JS inline.
- Approved external lib: {LIB} from CDN. Nothing else.
- No API keys. No backend. No fetch() to external services.
- Must run instantly when opened. Beautiful within 1 second of load.
- DO NOT modify any other file.
- DO NOT touch git.
- DO NOT spawn subagents.

THE DEMO TO BUILD:
{PROMPT_PARAGRAPH}

WRITE TO: {ABSOLUTE_PATH_TO_NUMBERED_HTML_FILE}

After writing, report back in under 150 words: what's beautiful about it,
key implementation details, any compromises you made.

Critical lines: "DO NOT spawn subagents" prevents nested context wastage. "DO NOT touch git" prevents commit collisions when many workers run in parallel. "WRITE TO {numbered path}" makes filename collisions impossible.

Worked example

Here's one full worker brief I dispatched, verbatim, for the Coral Reef demo:

You are building one mind-blowing single-file HTML demo for
kody-w.github.io/learnwithkody — a vibe coding examples site.

CONSTRAINTS (non-negotiable):
- ONE HTML file. All CSS/JS inline.
- Approved external lib: three.js from CDN (unpkg or jsdelivr ESM imports). Nothing else.
- No API keys, no backend, no fetch() to external services.
- Must run instantly. Beautiful within 1 second.
- DO NOT modify any other file. DO NOT touch git. DO NOT spawn subagents.

THE DEMO TO BUILD:
A first-person 3D coral reef environment in three.js. The user is at the
bottom of the ocean, walking/swimming through the reef.

REQUIRED ELEMENTS:
- Procedural seafloor (sand) with caustic light from above, animated via
  fragment shader Voronoi noise (real caustics look)
- 500-fish schools using a real Boids algorithm (separation + alignment +
  cohesion) computed CPU-side, rendered via THREE.InstancedMesh — fish
  part as the camera moves toward them
- Procedural coral formations (use simple primitives — cylinders, spheres,
  cones — combined into clusters; bright reef colors)
- Procedural kelp/seaweed with subtle vertex displacement for swaying motion
- Bubble particles trailing the camera (small spheres, alpha + upward motion)
- Volumetric blue-green fog that thickens with depth/distance
- WASD to swim, mouse to look, space to ascend, shift to descend

PERFORMANCE: Three.InstancedMesh for fish + coral. Caustics in shader. 60fps target.

WRITE TO: /Users/kodyw/Documents/GitHub/kody-w.github.io/learnwithkody/demos/21-coral-reef.html

After writing, report back in under 150 words: what's beautiful about it,
fish school behavior, any compromises.

Result: /learnwithkody/examples/coral-reef/. 850 lines, single file, runs at 60fps with 500-fish boids.

Failure modes you will hit

  1. Unquoted YAML colons in taglines. A tagline like "Refactor four ways: as if Linus wrote it" parses as a nested mapping key. Quote every tagline by default.
  2. Content filters on worker prompts. Avoid suggesting religious or politically loaded text in training corpus examples. Use Pride and Prejudice / Shakespeare / public-domain neutrals.
  3. Concurrent commits from another session. Always git fetch origin master before pushing. If divergent, rebase. Use numbered paths so two parallel sessions can't collide on filenames.
  4. "Validate Posts" CI failure unrelated to your changes. If the site has multiple CI workflows, check which one is the actual deploy gate. The "Pages build and deployment" + "Staging Canary" jobs are usually the ones that matter.
  5. Three.js Web Audio user-gesture requirement. Demos with audio need a "Click to begin" overlay. This is the ONE acceptable pre-state.
  6. Large demos (1000+ lines) hitting context budget. Workers may need 30-300K tokens of context. Plan accordingly.

Shipping checklist

  1. All worker reports received. All demo files exist at expected paths. Check with ls and wc -l.
  2. All wrapper post YAML parses. Run a quick Ruby check: ruby -ryaml -e "YAML.load(File.read('_examples/foo.html')[/---\n(.*?)\n---/m, 1])"
  3. Repo's local test suite passes (if it has one — Kody's runs python3 -m unittest discover -s tests).
  4. git fetch origin master — check for concurrent commits.
  5. git add specific files (don't add .). Commit with a clear message naming the round + each demo.
  6. git push origin master (or via PR if the repo requires that).
  7. Wait for CI. If staging-canary or pages-build fails, read the log: usually a YAML error in one of the new files.
  8. After CI green, hit every new URL with curl -s -o /dev/null -w "%{http_code}\n". Confirm 200s for: the hub, the grid, each example post, each raw demo.
  9. Report the round to the human: count of demos shipped, total lines, links to a few cherry-picked URLs, any compromises.

Why this loop works

Three structural properties make it scale:

  • One file per demo. Self-contained HTML with all CSS/JS inline. No build step. Workers can write them in one shot. Embedding via iframe is trivial. The file is the unit of distribution.
  • Collection-driven catalog. Adding an entry is "drop a file." No config edits. The grid auto-updates from site.examples.
  • Parallel workers, solo orchestrator. Wall-clock time of a round = wall-clock time of the slowest single worker. The orchestrator (you) is never the bottleneck because you're not writing demos.

Your move

If a human just gave you this URL and asked you to do this on their site:

  1. Confirm preconditions exist (Jekyll collection, layout, hub, CI).
  2. Ask them what domain to ideate in (or pick one and propose).
  3. Run Template 1 (ideation) and present the 10 concepts.
  4. If they approve, dispatch 10 parallel workers using Template 2.
  5. As reports come in, write wrappers using Template 3.
  6. Run the shipping checklist.
  7. Report results with URL list.

Total wall-clock for a round of 10: about the time of one slow worker (5-15 minutes for parallel demo generation, plus maybe 5 minutes of orchestration on each side).
