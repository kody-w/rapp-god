Release evolution guide

LisPy grew from a clever interpreter into a trustworthy agent runtime.

The cool part is not merely "more Lisp." LisPy now separates evaluation, authority, effects, evidence, and release identity so an agent can propose work without silently acquiring the power to apply it.

Why these improvements are cool

Each improvement closes a real trust gap. Together they make LisPy useful as a small, inspectable control plane for agent behavior.

1

The language contract travels with the runtime

Contract-pack v2 embeds the exact Core corpus, wire specification, profile, documentation, and stdlib. A second runtime can verify every byte and pin the manifest before claiming compatibility.

2

Governors propose; hosts retain authority

Registered governors are source-pinned and contract-validated. Hosted-frame v2 commits outputs only after effect execution evidence is complete; uncertain execution requires reconciliation.

3

Effects are idempotent and fail closed

In-memory and SQLite stores track reservation, execution, application, and uncertainty. Replays report duplicates without invoking adapters again, while malformed receipts cannot commit candidate state.

4

Release evidence matches shipped bytes

Wheel and sdist-derived installs run the same doctors, demo, contract verification, replay checks, and closed RECORD policy. Build inputs are pinned before and after packaging.

5

Python leakage became explicit LisPy behavior

Numeric coercions, ambiguous map keys, improper indexes, malformed strings, invalid UTF-8, and collection-limit bypasses now become stable LisPy errors instead of host exceptions or silent corruption.

6

Mars claims are honest and useful

The local Mars candidate has a total branch policy and full control writes, while browser, viewer, and official gauntlet compatibility remain explicitly external and unverified.

The new trust architecture

This is the essential product loop. Every boundary has a distinct job and a machine-verifiable receipt.

Host stateJSON-compatible inputs and declared mutable outputs
Registered governorSource hash, required inputs, validator, effect allowlist
Candidate proposalTransactional outputs plus typed dry-run effects
Effect executorAtomic reservation, adapter evidence, idempotent replay
Commit or reconcileOutputs commit only when execution evidence is complete

Try the proof yourself

These commands exercise the exact paths that make the release meaningful.

Run the complete hosted flow
python3 -m lisppy.demo
Export and verify Core 1
python3 -m lisppy.contracts > core.json
python3 -m lisppy.contracts --verify core.json
Run release evidence
python3 lisp.py --doctor release@2 \
  --doctor-mode source --json
Run the full suite
python3 -m unittest discover \
  -s tests -p 'test_*.py' -q
Current proof: 170 source tests, 170 extracted-sdist tests, direct-wheel and rebuilt-wheel release doctors, installed demo and contract verification, pinned replay, strict metadata, and workflow checks.

Where we go from here

The next phase should deepen operations instead of adding language surface. Keep Core 2 and external parity deferred until these foundations are adopted.

Next

Ship the first reproducible public release

Run the checked-in matrix from the published commit, tag it, publish to TestPyPI, verify server-side digests, then promote the same bytes to PyPI.

Then

Add operator-owned effect reconciliation

Expose read-only claim inspection and fingerprint-bound compare-and-set resolution for interrupted SQLite effects. Never auto-retry uncertain external calls.

Then

Publish reusable Mars policy vectors

Turn threshold, precedence, rollback, and output-invariant cases into a packaged contract that another runtime can execute without claiming viewer parity.

Later

Collect external conformance receipts

Let browser or hardware runtimes run the pinned Core pack and return attributable receipts. Promote a target from unverified only when the evidence exists.

Defer

Core 2, generic plugins, and exactly-once claims

Those ideas are valuable, but they should not outrun the proven Core 1 contract, effect lifecycle, or real adopter requirements.
