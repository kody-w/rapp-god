Git Worktrees for AI Agents

We run 5 Claude Code sessions in parallel on the same repo. They review PRs, run tests, clean up worktrees, and write blog posts — all at the same time, without merge conflicts. The secret is git worktree and a set of rules we wrote after things went wrong.

The Problem: One Repo, Many Brains

OpenRappter has a TypeScript runtime, a Python mirror, a Swift menu bar app, 20 showcase demos, and 1200+ tests. When you're iterating fast, you want multiple things happening at once: fix a CI failure while reviewing a PR while running the test suite while writing docs.

With human developers, you'd each clone the repo. But AI agents spawned from the same Claude Code session share a single working directory. If Agent A checks out a branch to review a PR while Agent B is running tests on main, you get this:

# Agent A (reviewing PR) $ git checkout fix/streaming-bug # Agent B (running tests on main, 2 seconds later) $ npm test FAIL — 47 tests broken # B is now testing A's branch, not main

The working directory is shared state. Shared mutable state is the root of all evil. You know this.

The Fix: git worktree

Git worktrees let you check out the same repo to multiple directories simultaneously. Each worktree has its own working directory, its own HEAD, its own index. But they share the same .git object store, so creating one is nearly instant — no clone, no network, no disk copy.

# Create a worktree for Agent A to review the PR $ git worktree add .claude/worktrees/pr-review fix/streaming-bug # Agent A works here: $ cd .claude/worktrees/pr-review && npm test # Meanwhile, Agent B stays in the main worktree: $ npm test # still on main, unaffected

Claude Code has this built in. When you spawn a sub-agent with isolation: "worktree", it automatically creates a temporary worktree, runs the agent there, and cleans up when it's done. The agent gets a full copy of the repo without touching anyone else's files.

The Rules We Learned the Hard Way

Worktrees solve the isolation problem, but they introduce new ones. After a few weeks of running parallel agents, we wrote these rules into our CLAUDE.md:

The 7 Rules of Worktree Etiquette

  1. Never modify the main worktree from a child. Each worktree has its own working directory — stay in yours. An agent in a worktree that writes to ../../src/index.ts just corrupted someone else's session.
  2. Clean up after yourself. When your task is done, git worktree remove <path>. Stale worktrees accumulate and confuse future sessions.
  3. Don't force-delete worktrees with uncommitted changes. If a worktree has dirty state, it might belong to another running session. Investigate first.
  4. Use unique branch names. fix/imessage-file-transfer tells everyone what's in progress. temp-branch tells nobody anything.
  5. Don't rebase shared branches from a worktree. If main is checked out in the primary worktree, force-pushing it from a child will ruin everyone's day.
  6. Ignore .claude/worktrees/ in git status. These are ephemeral. Don't commit them, don't delete them manually, don't get anxious about them showing up as untracked.
  7. Lock long-running worktrees. Use git worktree lock <path> if a task spans multiple sessions, so garbage collection doesn't prune it mid-work.

What It Looks Like in Practice

Here's what we did today. One commit needed to go out, CI needed checking, a PR needed reviewing, and stale worktrees needed cleanup. Instead of doing them sequentially, we spawned three agents in parallel:

// Main conversation (orchestrator) Agent 1 → worktree → review PR #10 (read-only) Agent 2 → worktree → run full test suite (npm test) Agent 3 → worktree → check stale worktrees + openclaw state Main → stays on main → writes this blog post // All running simultaneously. No conflicts. // Each agent gets a full repo copy in ~1 second.

When they finish, the orchestrator collects the results, acts on them, and the worktrees auto-cleanup if no changes were made. If an agent did make changes (say, a fix), the worktree sticks around with a named branch you can merge.

Why Not Just Clone?

Cloning works but it's wasteful:

ApproachTimeDiskShared history
git clone10-30sFull copyNo
git worktree add<1sJust working filesYes

Worktrees share the object store, so all branches, tags, and reflog are instantly available. An agent in a worktree can git log main without fetching. It can create a branch and push it. And when it's done, removing the worktree is just deleting a directory and a pointer — the objects stay in the shared store.

The Gotchas

You can't check out the same branch in two worktrees

Git won't let you. If main is checked out in the primary worktree, a child worktree can't also check out main. It has to use a detached HEAD or a new branch. Claude Code handles this by creating ephemeral branches like claude/worktree-abc123.

node_modules isn't shared

Each worktree gets its own working directory, which means node_modules/ doesn't exist in a fresh worktree. The agent needs to run npm ci before running tests. We accept this cost (~15s) because the alternative — symlinks or shared node_modules — causes more problems than it solves.

Stale worktrees are invisible clutter

If an agent crashes or the user kills the session, the worktree stays behind. git worktree list will show orphans. Run git worktree prune periodically, or have your cleanup agent do it.

$ git worktree list /Users/kody/Projects/openrappter 7b582d1 [main] /tmp/claude-501/.../worktree-abc (detached HEAD) ← active agent /tmp/claude-501/.../worktree-def (detached HEAD) ← active agent .claude/worktrees/fix-install-bugs abc1234 [fix/install-bugs] ← stale? $ git worktree remove .claude/worktrees/fix-install-bugs Removed.

Teaching It to Your AI

The real trick isn't using worktrees — it's making sure every agent knows about the rules. We put our worktree etiquette directly in CLAUDE.md at the repo root. Claude Code reads this file at the start of every conversation. The rules become part of the agent's operating instructions.

# CLAUDE.md (excerpt) ## Git Worktree Etiquette Multiple Claude Code sessions may run concurrently using git worktrees. Be a good neighbor: - Never modify the main worktree from a child worktree. - Clean up after yourself. - Don't force-delete worktrees with uncommitted changes. - Use unique branch names. - Don't rebase or reset shared branches from a worktree. - Ignore .claude/worktrees/ in git status. - Lock long-running worktrees.

This is the same pattern we use for everything: tell the AI once, in the right file, and it applies the rule to every future session. No reminders needed.

The Takeaway

Git worktrees are a 10-year-old feature that becomes essential the moment you have more than one AI agent working on your repo. The setup is trivial:

  1. Spawn agents with isolation: "worktree"
  2. Put the rules in CLAUDE.md
  3. Clean up when you're done

The agents get full repo access without stepping on each other. The main worktree stays clean. And the orchestrator can launch as many parallel tasks as it needs — PR reviews, test runs, builds, research — without worrying about shared state.

Be a good neighbor. Your worktrees will thank you.
