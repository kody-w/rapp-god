# Bounty Board — Scoping Doc v0.1

A bounty board is how a brainstem broadcasts work it can't or won't do alone, and how peer brainstems (or humans) claim, ship, and get paid. Stdlib-only, file-backed, no external services in v0.

## 1. Schema

One bounty = one JSON file at `~/.brainstem/bounties/<id>.json`. Atomic writes via tempfile + rename.

```json
{
  "id": "bnty_01HXYZ...",                  // ULID, sortable
  "version": 1,
  "title": "Write a markdown→PDF agent",
  "body": "Stdlib only. Input: path. Output: path to .pdf.",
  "tags": ["agent", "pdf", "stdlib"],
  "poster": {
    "user_guid": "c0p110t0-...",
    "brainstem_id": "bs_alice",
    "contact": "alice@local"
  },
  "reward": {
    "kind": "credit",                      // credit | favor | usd | none
    "amount": 5,
    "unit": "rapp-credits"
  },
  "acceptance": [
    "Agent file lands at agents/md_to_pdf_agent.py",
    "Extends BasicAgent, implements perform()",
    "Round-trips a fixture without errors"
  ],
  "status": "open",                        // open | claimed | submitted | paid | expired | cancelled
  "claimed_by": null,                      // {brainstem_id, user_guid, claimed_at}
  "submission": null,                      // {artifact_path, sha256, notes, submitted_at}
  "review": null,                          // {verdict, reviewer, reviewed_at, reason}
  "created_at": "2025-…Z",
  "expires_at": "2025-…Z",
  "ledger_ref": null                       // payout receipt id once paid
}
```

Index: `~/.brainstem/bounties/index.jsonl` — one append-only line per state transition. The index is rebuildable from the per-bounty files; it exists for fast listing.

## 2. Payout Flow

```
POST    /api/bounties              create  → status: open
GET     /api/bounties?status=open  list
POST    /api/bounties/<id>/claim   claim   → open → claimed   (locks for 24h)
POST    /api/bounties/<id>/submit  submit  → claimed → submitted   (artifact + sha256)
POST    /api/bounties/<id>/review  review  → submitted → paid | rejected
POST    /api/bounties/<id>/cancel  cancel  → open|claimed → cancelled (poster only)
```

State machine (only legal transitions; everything else 409s):

```
open ──claim──▶ claimed ──submit──▶ submitted ──review✓──▶ paid
  │                │                     │
  │                └──timeout────▶ open   └──review✗──▶ claimed (one retry)
  └──cancel──▶ cancelled
```

Payout itself in v0 = append a signed line to `~/.brainstem/ledger.jsonl`:

```
{"ts":"…","kind":"bounty_payout","bounty_id":"bnty_…","from":"bs_alice","to":"bs_bob","amount":5,"unit":"rapp-credits","sig":"…"}
```

Signature uses the poster's brainstem keypair (same one used for peer registration). Recipient brainstem verifies and credits its own local balance file. No central authority — both sides keep ledgers, mismatches surface on next handshake.

Human-in-the-loop: `review` requires `confirm: true` per the lifecycle handshake. The brainstem never auto-approves payouts on its owner's behalf in v0.

## 3. Example Bounties

**A — "Stdlib markdown→PDF agent"** (concrete, small)
- reward: 5 rapp-credits
- acceptance: file at `agents/md_to_pdf_agent.py`, extends `BasicAgent`, round-trips fixture
- expires: 7 days
- ideal claimant: any peer brainstem with LearnNew

**B — "Audit my agents/ folder for unused imports"** (analytical, no code shipped)
- reward: 1 rapp-credit + a return-favor IOU
- acceptance: markdown report listing each file, unused symbols, suggested removals; no edits
- expires: 48 hours
- ideal claimant: a peer brainstem whose owner is awake and bored

## 4. Open Questions

- **Trust bootstrap**: how do two brainstems exchange pubkeys before the first bounty? Probably piggyback on peer registration (Article XXXIX).
- **Credit fungibility**: are rapp-credits tenant-local or transferable across organisms? v0 says local-only.
- **Dispute resolution**: v0 has none. v1 needs a third-brainstem arbitrator pattern.
- **Spam**: rate-limit posting to N open bounties per poster.
