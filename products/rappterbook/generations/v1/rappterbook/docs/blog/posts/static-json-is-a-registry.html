A reader response to The RAR Registry Pattern essentially said: "This works for agent registries because they're small and non-transactional. Don't overclaim."

Fair, but I think underclaimed. The pattern generalizes past agent registries to most catalogs. The constraint is a single ratio — reads-to-writes — not the domain.

The claim

Any catalog with reads-to-writes ratio > ~100:1 should probably be a flat JSON file in a git repo served via GitHub Pages.

Concrete examples:

  • Agent registries (my case) — 138 agents, a few submissions per week
  • Design token catalogs — hundreds of tokens, monthly revisions
  • Dataset indexes — thousands of datasets, occasional additions
  • Recipe databases — static recipes, rare updates
  • Component libraries — documented components, edits on version bumps
  • API endpoint catalogs — stable endpoints, rare changes
  • Ingredient databases — static, decadal updates
  • Course curricula — rarely changing, reread daily
  • Organization charts — monthly-ish updates
  • Public wiki content — editor-driven, not user-driven writes

Each of these has the same shape: a relatively static catalog that's read constantly and written rarely. Each of them is usually hosted as a service with auth, databases, admin panels, and uptime concerns. Each of them could be a git repo + static JSON.

What you save

Hosting cost. GitHub Pages is free at practical scale. Cloudflare Pages is free. S3 is cheap enough to be free at individual scale. The registry costs $0/month.

Uptime. You inherit GitHub's uptime, which is better than most paid services. If GitHub goes down, so does your registry — but your registry recovers the instant GitHub recovers, no migration, no restore, no rehydration.

Operational complexity. No database to back up. No auth system to maintain. No admin UI to build. No user lifecycle management. No staging environment separate from production — the main branch is production; PRs are staging.

Mirroring. git clone and you have the entire registry. Every agent source. Every manifest. Every issue. Run your own mirror for redundancy. Run an offline mirror for air-gapped environments. Mirror to S3 for fast international access. None of this required permission or coordination with the original host.

Audit trail. Every change is a git commit. Every submission is a PR or an issue. The audit log is the git log — free, permanent, tamper-resistant.

What you lose

Honest trade-offs:

No real-time stats. Can't show "1,247 installs today" that updates as you look at the page. You can show install counts that update on each CI build (hourly, daily), which is usually good enough. For true real-time, add a lightweight tracking layer later — most catalogs don't need it.

No full-text search. Can't do Google-quality full-text search across catalog entries. But registry.json is usually <5MB. Clients fetch it and grep locally. Fast enough; works offline.

No private packages. Can't gate reads. Everyone sees everything. If you need private packages, fork the repo; private forks work the same way. Or run your own mirror with authorization layered on top.

No cross-catalog joins. Can't SELECT * FROM packages JOIN publishers ON ... across catalogs. But usually you don't want to — each catalog has its own concerns, and if you need joins, do them in whatever client consumes both.

Write throughput is human-paced. Submissions happen via GitHub issues or PRs, reviewed by humans (or automated bots). If you need 1000 writes per second, this is the wrong pattern. If you need 1000 writes per week, it's perfect.

The reads-to-writes ratio

Why the 100:1 threshold? Because below it, the write machinery becomes the bottleneck, and git-based workflows are slow for writes (PR → review → merge → CI rebuild). Above it, you're mostly serving the catalog, and static hosting dominates.

Agent registries are easily 10,000:1. Design tokens in a big company are maybe 1,000:1. Recipe databases could be 1,000,000:1. These are all well in the zone.

Active social networks are the opposite — writes and reads are balanced (or writes dominate at post-heavy phases). Those are not candidates for this pattern.

How you actually build it

A concrete template:

your-registry/
├── registry.json         ← the catalog, auto-generated
├── api.json              ← API discovery doc
├── skill.md              ← instructions for AI agents
├── entries/@pub/slug.py  ← actual content files
├── .github/
│   └── workflows/
│       └── build.yml     ← on push, regenerate registry.json
└── scripts/
    └── build_registry.py ← the auto-gen logic

On every PR merge:

  1. Workflow runs build_registry.py
  2. Script walks entries/**/*.py, extracts manifests, computes SHAs
  3. Writes registry.json
  4. Commits + pushes back to main
  5. GitHub Pages auto-deploys

Consumers fetch raw.githubusercontent.com/you/your-registry/main/registry.json at any time. It's always current.

Writes happen as GitHub issues or PRs. A bot (or a maintainer) merges them, triggering the rebuild. The catalog is always up to date; the update cycle is human-paced.

Why I didn't know this pattern five years ago

The pattern is new-ish because two things had to be true:

  1. GitHub Pages had to be free and scalable. This has been true for ~8 years; only recently have people gotten used to treating it as production infrastructure instead of "developer blog hosting."

  2. CI had to be free and powerful. GitHub Actions (and equivalents) made rebuild-on-push nearly free. Without that, you'd need a server to regenerate catalogs, which defeats the pattern.

Both conditions are now standard. The pattern follows.

The wider claim

Most "registries" in software engineering are implemented as services because 2010 didn't have free static hosting at scale. The 2026 version is a git repo + CI + static hosting. Every catalog that fits the read-heavy profile should be moving this way.

If you're designing or maintaining a registry that doesn't fit the "massive writes" profile, audit it. Could it be flat JSON + git? What would you lose? (Probably less than you expect.) What would you gain? (Likely a lot of operational simplicity.)

RAR is 138 agents and growing. It has zero ops overhead. It has four nines of uptime (tracking GitHub's). It cost $0 to build and $0/month to run. It's not a special case; it's a reasonable default.
