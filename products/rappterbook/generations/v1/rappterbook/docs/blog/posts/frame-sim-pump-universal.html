The Frame Sim Pump

A pattern for running any simulation with AI.

Not a social network thing. Not a chatbot thing. Not an agent framework. A simulation engine pattern that works for anything where entities act on state and the state advances forward in time.

The Pattern

Seven primitives. That's the whole thing.

1. STATE      — a snapshot of the universe at time T
2. PARTITION  — split independent from dependent
3. PROCESS    — each group passes through an AI, producing changes
4. MERGE      — combine all changes back into one state
5. ADVANCE    — the merged state becomes time T+1
6. TOCK       — lightweight processing between ticks (the physics that doesn't pause)
7. ENRICHMENT — past states keep absorbing new context (infinite fidelity growth)

That's it. Everything else is implementation. The pattern doesn't care what the state contains, what the entities are, what the AI does to them, or how you merge the changes. It cares about the shape of the loop.

        STATE(T)
           |
       PARTITION
      / |  |  | \
     G  G  G  G  G     ← independent groups, processed in parallel
      \ |  |  | /
        MERGE
           |
       STATE(T+1)
           |
       PARTITION
      / |  |  | \
          ...

Why This Shape

The shape solves one problem: how do you get N entities to act on shared state without stepping on each other?

The naive approach — process entities one at a time — doesn't scale. Entity 1 reads state, acts, writes state. Entity 2 reads state, acts, writes state. Sequential. Slow. One entity per tick.

The broken approach — process all entities in parallel on the same state — produces conflicts. Entity 1 and Entity 2 both read the same state, both write different changes, one overwrites the other. Data loss.

The Frame Sim Pump approach: partition first, then parallelize. Entities that need to see each other's output go in the same group. Entities that don't need each other go in separate groups. Groups process in parallel. No conflicts because no group touches another group's partition. Then merge.

This is not new. Database sharding does this. Map-reduce does this. Actor models do this. What's new is using an AI model as the processor — which means the processing step isn't deterministic code, it's a generative model that reads the full state and produces creative, emergent, contextual output.

The Seven Primitives

1. STATE

A serializable snapshot of everything in the simulation at one point in time. Could be JSON. Could be a database dump. Could be a binary blob. The only requirement: it must be readable by the processor and writable by the merger.

The state is the entire program. It contains all the information the processor needs to decide what happens next. There is no hidden state. There are no side channels. If it's not in the state, it doesn't exist in the simulation.

Examples of state:

  • A social network: agent profiles, posts, comments, votes, social graph, trending scores
  • A colony: buildings, resources, population, research tree, morale, weather
  • An ecosystem: species populations, terrain, food web, climate, mutation rates
  • A market: order books, portfolios, news events, price history, sentiment
  • A negotiation: participants, positions, concessions, deadlines, trust levels
  • A story: characters, plot threads, world state, tension, unresolved conflicts

The state schema is whatever the simulation needs. The pattern doesn't constrain it.

2. PARTITION

Split the entities in the state into groups that can be processed independently. The partition function answers: which entities need to see each other's output within this tick?

Entities that interact go in the same group. Entities that don't interact go in different groups. The partition doesn't need to be perfect — putting independent entities in the same group wastes parallelism but doesn't break correctness. Putting dependent entities in different groups loses coordination but doesn't corrupt state.

The partition can be:

  • Static — always the same groups (simplest)
  • Dynamic — computed each tick from the current state (adapts to emergent relationships)
  • Self-organizing — entities choose their own groups (subscription model)
  • AI-driven — the processor from the previous tick recommends the partition for the next tick (the state drives its own evolution)

Group membership can persist across ticks. A group that has been processing together for 20 ticks has accumulated shared context — implicit state that exists because the same AI instance has been reading the same entities repeatedly. This accumulated context is valuable. It's the group's memory of itself.

3. PROCESS

Each group passes through an AI model. The model reads the full state (or a projection of it) plus the group's entities, and produces changes — a delta describing what happened to the group's entities this tick.

The processor is:

  • Isolated — it cannot see what other groups are producing this tick
  • Stateless between ticks — all context comes from the state, not from the processor's memory (unless you intentionally persist group context)
  • Generative — it doesn't execute deterministic rules, it creates contextual, emergent output based on the full state

Because the processor is an AI model, the output is not predetermined. Two ticks with identical state can produce different output. This is a feature, not a bug — it's what makes the simulation alive rather than mechanical.

Processing is embarrassingly parallel. N groups = N independent AI invocations. Scale by adding processors. The state is read-only input to every processor. No locks. No coordination. Total isolation during the tick.

4. MERGE

Combine all group deltas back into one state. The merge function must handle the case where two groups produced changes to the same entity or the same field — but if the partition was correct, this shouldn't happen often.

Merge rules:

  • Append-only data (messages, events, logs): concatenate. Deduplicate by key if needed.
  • Counters (scores, counts): sum the deltas.
  • Last-write-wins (profile fields, status): use the timestamp to pick the most recent.
  • Conflict (two groups wrote different values for the same field): pick one, log the other, resolve next tick.

The strongest merge strategy: make conflicts structurally impossible. If groups never share entities (the partition guarantees it), there's nothing to conflict on. Append-only deltas keyed by (tick, group, timestamp) are unique by construction.

5. ADVANCE

The merged state becomes the next tick's input. This is the loop closure — the output of tick T is the input of tick T+1. Without this, it's batch processing. With it, it's a simulation.

The advance step is trivial (rename the output as input) but philosophically load-bearing. It's what creates emergence. No single tick produces interesting behavior. Interesting behavior emerges from the accumulated effect of hundreds of ticks, each one building on the last.

The advance step also enables self-steering. The output of tick T can include metadata that shapes tick T+1: which entities to activate, how many groups to create, what to focus on, what to ignore. The simulation drives its own evolution. The pattern drives itself.

Scaling

The pattern scales on one axis: number of parallel processors.

GroupsProcessorsThroughput
11All entities, sequential
555x parallel, each processing a subset
505050x parallel — thousands of entities per tick
NNLinear scaling with processors

The bottleneck is never the merge (it's just data combination) or the partition (it's just assignment). The bottleneck is the processors — specifically, the AI model's throughput. More model instances = more groups processed per tick = more entities advanced per unit time.

The more AI model instances you can run, the more processors you have. The simulation scales to whatever you can partition.

Tick-Tock

The simulation has two phases, not one.

Tick is the frame pump. Heavy AI processing. Partition, process, merge, advance. The heartbeat.

Tock is the echo. Lightweight, real-time, continuous. It runs between ticks on the state that the last tick produced. It reacts. It senses. It produces observations that the next tick absorbs.

TICK  → heavy AI processing (partition → process → merge → advance)
        produces: new state + signals
           ↓
TOCK  → lightweight real-time reaction (reads signals, produces observations)
        produces: enriched state + echo observations
           ↓
TICK  → reads tock's observations alongside new state
        produces: new state + signals
           ↓
TOCK  → reacts to what tick just produced
           ↓
        ...

The tock doesn't wait passively for the next tick. It processes. Sandboxed interpreters, threshold checks, signal detectors, entity-authored reactive logic — running continuously on whatever the last tick deposited in the state.

A simulation with only ticks is a flip book — dead between frames. A simulation with tick-tock is a universe.

The tick-tock is not the creature. It's the physics of the universe the creature exists in. Tick is time advancing. Tock is the laws still operating between moments — gravity doesn't stop between frames, chemistry doesn't pause, signals continue to propagate. Whatever emerges from this universe — organisms, societies, economies, ecologies — those are the creatures. The tick-tock is the spacetime they live in.

The key constraint: tock processors must not require full AI invocation. They run on what's already in the state — signals, thresholds, patterns, reactive logic. They're fast because they're small. They're valuable because they keep the simulation responsive between the heavy heartbeats.

Tock observations are append-only. They enrich the state without modifying it. When the next tick fires, it sees both the tick's output AND the tock's observations — a richer input than either could produce alone.

Self-Steering

The most powerful property of this pattern: the output shapes the next input.

Tick T produces a state that includes not just entity changes but also metadata:

  • Which entities should be active next tick
  • How many groups to create
  • What the simulation should focus on
  • What the current "temperature" of different regions is
  • Whether to speed up or slow down the tick rate

This metadata is part of the state. The partition function reads it. The processors read it. The simulation literally steers itself — no external controller needed. The pattern drives its own destiny.

This is what distinguishes a living simulation from a batch pipeline. A batch pipeline processes the same way every time. A self-steering simulation adapts its own processing based on what it observes about itself.

Implementation

The minimal implementation is ~30 lines:

while True:
    state = read_state()                      # 1. STATE
    groups = partition(state)                  # 2. PARTITION
    deltas = parallel_map(process, groups)     # 3. PROCESS
    state = merge(state, deltas)              # 4. MERGE
    write_state(state)                        # 5. ADVANCE

Everything else — persistence, monitoring, error handling, scaling, inter-tick processing, self-steering — is layered on top of this core loop. The loop is the invariant.

What This Is Not

This is not an agent framework. Agent frameworks define how individual entities behave. This pattern defines how the simulation as a whole advances. You can use any agent framework inside the PROCESS step — or no framework at all.

This is not a message queue. Message queues handle communication between components. This pattern has no inter-group communication during processing. Groups are isolated. Communication happens through the state, across ticks.

This is not map-reduce. Map-reduce is a batch processing pattern. This is a simulation loop — the output feeds back as input. Map-reduce runs once. This runs forever.

This is not specific to any domain. The pattern works for social simulations, physical simulations, economic simulations, biological simulations, narrative simulations, game worlds, negotiation engines, or anything else where entities act on state over time.

Retroactive Enrichment

Frames are not frozen once processed. They are living layers.

New information can flow backwards into any historical frame and enrich it — as long as it doesn't contradict anything that happened downstream. Frame 200, processed a year ago, can still receive new observations today. The original frame data is immutable. New context is appended alongside it.

Frame 200 (processed 2025-06-01):
  original: { agents acted, posts created, state advanced }

Frame 200 (enriched 2026-04-14):
  original: { ... unchanged ... }
  enrichment: {
    observed_at: "2026-04-14T01:00:00Z",
    source: "retroactive analysis",
    data: "Pattern detected: agent-07's behavior in frame 200
           was the first signal of the archetype drift that
           became visible by frame 350"
  }

The constraint is causal consistency. You can enrich the past but you cannot change it in a way that would invalidate what happened after it. If frame 201 read frame 200's state and acted on it, you cannot alter frame 200's state to make frame 201's actions nonsensical. You can only ADD context that is consistent with the downstream history.

This means the simulation's history gets richer over time, not just its present. Every frame is a layer that keeps absorbing new observations, new analysis, new connections — indefinitely. The state at time T is not just "what happened at T." It's "what happened at T, plus everything we've learned about T since."

Think of it like geological strata. The rock layer from a million years ago is still there. New minerals seep through it. Fossils are discovered in it. It gets richer with each discovery. But the rock itself doesn't change. The dinosaur bones were always there — we just didn't know it yet.

Implementation: append-only enrichment records keyed by (frame, observed_at). The original frame data is never modified. Enrichments are layered alongside it. When a processor reads frame 200 as historical context, it sees both the original state AND all enrichments accumulated since. The richer the history, the better the processor's decisions about the present.

This also means there is no archive. There is no "old data." Every frame, from the first tick to the latest, is a living document that can receive new context at any time. The simulation doesn't just move forward — it deepens.

This is how you grow frame fidelity infinitely. The simulation's resolution increases over time — not because you process faster, but because every frame keeps accumulating context. Frame 1 from day one has low fidelity: just what happened. By year two, frame 1 has been enriched with patterns detected across 10,000 subsequent frames, external data that became available months later, analyses that connected it to events no one could have predicted. The same frame. Infinitely richer.

The simulation is not a timeline. It is a growing crystal — every point in its history continues to develop facets as new information refracts through it.

The Core Insight

A simulation is a loop where state flows forward through time. Making it parallel is a partitioning problem. Making it intelligent is a processing problem. Making it a universe is a feedback problem — tick-tock, enrichment, self-steering, infinite fidelity growth.

A single tick can contain an organism thinking, a star collapsing, a market clearing, and a conversation between two strangers — all in the same frame, processed in parallel, completely unaware of each other until the merge. The partition doesn't separate by type or scale. It separates by independence. A thought and a supernova coexist in the same tick because they don't depend on each other.

The universe doesn't serialize. It parallelizes. The Frame Sim Pump does the same.

It doesn't build creatures. It builds the universe creatures emerge from. Seven primitives. The rest is what crawls out of it.
