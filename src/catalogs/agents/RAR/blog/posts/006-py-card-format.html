One File, Two Modes

A core discovery in the RAPP ecosystem: an agent has two natural states. The bare agent (.py) is just code and manifest — the minimum deployable unit. The agent card (.py.card) is the complete package — code, manifest, and the full trading card shell.

The Shell Metaphor

Think of it like a hermit crab. The agent (.py) is the creature. The card (__card__ dict) is the shell. The creature works fine without the shell — it can still perform, still be imported, still run. But with the shell, it has visual identity, stats, abilities, art, and the ability to be collected, traded, and displayed.

# slug.py.card — the complete package

__manifest__ = { ... }  # Agent identity (who am I?)

__card__ = {             # Card shell (how do I look?)
    "name": "Display Name",
    "title": "The Subtitle",
    "mana_cost": "{2}{U}{B}",
    "colors": ["U", "B"],
    "type_line": "Creature — Agent Type",
    "rarity": "mythic",
    "power": 6,
    "toughness": 4,
    "abilities": [...],
    "flavor_text": "Lore text.",
    "avatar_svg": "<svg>...</svg>",
    "set_code": "HOLO",
    "artist": "Howard",
}

class MyAgent(BasicAgent):
    def perform(self, **kwargs): ...

Shedding

Shed = strip the __card__ dict from a .py.card. What's left is a valid .py agent.

Why shed? When you don't need the visual identity. When you're deploying to production and want minimal file size. When the agent is running headless in a pipeline.

Re-Shelling

Re-shell = add __card__ back to a bare .py from the registry or holo_cards.json. The card data is reconstructed from the agent's first public publish point — the hatching moment.

Why re-shell? When you have a bare agent and want to display it as a card. When you're building a local binder and need the full visual package. When you're trading cards and need the shell to travel with the code.

Registry Integration

The build_registry.py scanner was updated to:

  1. Scan for both *.py and *.py.card files
  2. Prefer .py.card when both exist for the same agent
  3. Extract __card__ via AST parsing (same as __manifest__)
  4. Include _has_card and _card fields in the registry entry

The first two .py.card files in the repository are under @borg — the Borg agent and the CardSmith agent, both with original card data from the HOLO set.

Compatibility With CardSmith

The __card__ schema matches the CardSmith agent's _CARD_DATABASE format exactly. Same fields, same types, same structure. A card generated by CardSmith can be embedded directly as a __card__ dict. A __card__ dict can be read by CardSmith as if it were in the database.

This is the interchangeability the system was designed for. The card is the agent. The agent is the file. The file is the package.
