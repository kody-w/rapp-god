The Extension Chain Is the Manifest

Most packaging systems need a separate manifest to describe their contents. npm has package.json. Docker has Dockerfile. Kubernetes has YAML files. RAPP has... the file extension.

agent.py                         → bare agent
agent.py.card                    → agent + card shell
agent.py.card.Genesis            → agent + card shell + deck membership

Each layer of the extension chain adds metadata without changing the core content. Strip a layer, you lose that metadata. Add a layer, you gain it. The agent code is always intact at every level.

Why the Extension Chain Works

Traditional packaging creates a separation between content and metadata. The content lives in one file, the metadata in another. This creates synchronization problems: what happens when the manifest says one thing and the file says another?

RAPP's answer: there is no separation. The file IS the manifest. The extension IS the deck membership. Everything travels together.

LayerWhat It AddsReversible?
.pyAgent code + __manifest__This is the base
.card__card__ dict (visual identity, stats, art)Yes — strip to get bare .py, re-add from registry
.DeckNameDeck membershipYes — rename to remove deck association

Deck Import/Export

To share a deck: collect all files ending in .py.card.DeckName for that deck name. To import: drop them into any RAPP binder. The deck name, card data, and agent code all travel together in a single file per agent. No zip files. No manifests. No configuration.

# Export: find all Genesis deck cards
ls agents/@borg/*.py.card.Genesis

# Import: copy into another binder
cp *.py.card.Genesis ~/my-binder/decks/Genesis/

Runtime Hotloading

This is where the deck extension gets powerful. In a RAPP Brainstem runtime, the agents/ folder stays pristine — only bare .py files live there. These are the base agents that are always available.

Deck files live in a separate decks/ directory, organized by deck name:

brainstem/
  agents/                          ← always loaded (base set)
    basic_agent.py
    borg_agent.py
  decks/                           ← hotloadable
    Genesis/
      borg_agent.py.card.Genesis
      anvil_agent.py.card.Genesis
    Frontier/
      experiment_agent.py.card.Frontier

When the user selects a deck from the frontend, the runtime hotloads the agents from that deck. Switch decks, switch active agents. No restart. No config change. Just like selecting a different AI model to drive the conversation — but for the agent set.

The Model Switcher Analogy

In the RAPP Brainstem, users already choose which AI model powers the system (cloud GPT, local Gemma, etc.). The deck hotloader works the same way but for agents:

SwitcherWhat It ControlsLives In
Model selectorWhich AI model drives responses.env / config
Deck selectorWhich agents are activedecks/DeckName/

Both are user-facing, real-time switches. Both require zero restart. Both keep the core system clean.

Keeping agents/ Pristine

The agents/ directory is git-tracked and CI-validated. Injecting deck files into it would create noise. The decks/ directory is user-managed and gitignored by default. This separation means:

  • agents/ is the source of truth for what agents exist
  • decks/ is the user's curation of which agents are active
  • The registry builder scans agents/
  • The runtime loader scans decks/ for the active deck
  • No conflict between the two

Field Note

The realization that triggered this design: file extensions are metadata that survives every operation. You can copy, move, rename, zip, email, airdrop, or sneakernet a file — and the extension survives. It's the most portable metadata format possible. Using it for deck membership means decks are as portable as the files themselves. No database. No manifest. No sync. Just filenames.
