The Last Missing Piece

The RAPP Brainstem has always had three components: an agent deck (the agents you've loaded), a dispatch loop (routing user intent to the right agent), and an LLM layer (understanding what the user wants). The local-first version runs Ollama on localhost:11434. It works. But it requires a local GPU, a model download, and a terminal.

The Virtual Brainstem solved agents (transpile to JS) and dispatch (LisPy VM). But without an LLM, it's a REPL, not a brainstem. You have to know which operation to call and type it as an S-expression. That's a developer tool, not a product.

GitHub Models closes the gap. It's an OpenAI-compatible inference API authenticated with your GitHub token. The same OAuth flow that lets you vote on agents now lets you chat with them.

The Architecture

User types: "what's trending on hacker news?"
       |
       v
  GitHub Models API (gpt-4o-mini)
  with function definitions for each agent
       |
       v
  tool_call: hacker_news({limit: 10})
       |
       v
  Browser executes: agentHackerNews({limit: 10})
  -> fetch HN Firebase API -> return stories
       |
       v
  GitHub Models API (follow-up with tool result)
       |
       v
  "Here are today's top stories on Hacker News: ..."

The LLM never touches the agent data directly. It decides which agent to call and what arguments to pass. The agent runs locally. The LLM formats the result for the user. This is the same separation of concerns as the local brainstem — the LLM reasons, the agents execute.

Shared Authentication

The RAPP ecosystem already has GitHub OAuth for voting and reviews. The token is stored as rb_access_token in localStorage, shared with the RAPPterBook. The Virtual Brainstem reads the same token. Sign in once, use it everywhere.

The OAuth flow redirects to GitHub, exchanges the code via a Cloudflare Worker, and stores the token. The same token authenticates both the GitHub API (for user info) and the GitHub Models API (for inference). No second login. No API key management.

Model Selection

GitHub Models exposes different models depending on your account tier. The Virtual Brainstem fetches the available models on sign-in and populates a dropdown. Your selection persists to localStorage.

The models endpoint returns Azure ML registry URIs like azureml://registries/azure-openai/models/gpt-4o/versions/2. The UI extracts the clean model name (gpt-4o) and deduplicates. If the endpoint is unavailable, it falls back to known defaults.

What This Pattern Enables

The local brainstem requires: Python, Ollama, a GPU, a model download, and a running server. The virtual brainstem requires: a browser and a GitHub account.

The agent behavior is identical. The chat experience is identical. The dispatch pattern is identical. The only difference is where the LLM runs. That's the virtualization — the same brainstem architecture, fully hosted in your browser tab and someone else's GPU.

When you outgrow the free tier, switch back to local. The agents don't change. The conversation pattern doesn't change. The brainstem is the brainstem, whether it runs on your machine or in the cloud. That's what "local-first" actually means: local is the default, but not the requirement.
