The Problem: 128 Agents Nobody Can Run

RAPP had 128 agents in the registry. Every one of them was a single .py file. You could browse them, read them, fork them. But to run one, you needed Python installed, the brainstem runtime configured, and a terminal. That's three barriers between a user and an agent.

The single-file principle gave us portability. But portability without execution is just a museum.

What If the Browser Was the Runtime?

The Virtual Brainstem is a single HTML file — virtual-brainstem.html — that contains a complete agent execution environment. No server. No install. No Python. Open a URL, pick an agent from the sidebar, and it runs.

The architecture has five layers, all in one file:

unrapp.jsPython-to-JavaScript transpiler for the agent subset~800

ui.jsSidebar, terminal, REPL, drag-and-drop~300
chat.jsConversational AI with GitHub Models + built-in agents~400

Two Modes, One File

The Virtual Brainstem ships with two modes:

Chat mode is the default. Sign in with GitHub, type naturally, and the AI routes your message to the right agent. "What's trending on Hacker News?" calls the HackerNews agent. "Remember I prefer dark mode" calls ManageMemory. The brainstem pattern — LLM reasoning over a deck of agents — works identically to the local-first version, except the LLM is GitHub Models and the agents are JavaScript functions running in your browser tab.

Lab mode is the agent workbench. It loads the full registry, lets you click any agent, transpiles its Python source to JavaScript on the fly, and executes it in a LisPy VM. You can interact via a REPL, drop local .py files, and run the built-in test suite.

The Pipeline

When you select an agent from the registry, this happens in under 100ms:

  1. Fetch — one GET to raw.githubusercontent.com downloads the agent's .py file
  2. Transpile — the Python source is converted to JavaScript functions via a 5-stage pipeline
  3. Compile — each function is wrapped in a closure with Python builtins (len, sorted, round, .get()) shimmed as a PY namespace
  4. Inject — compiled functions and data objects are loaded into a fresh LisPy environment
  5. Dispatch — a LisPy perform function routes operations to the right method

The result: any agent with synthetic data runs natively in the browser. No CORS proxy. No WASM. No bundler. Just JavaScript doing what JavaScript does.

Why a Lisp?

The LisPy VM exists because agent dispatch is an S-expression problem. (perform (make-dict "operation" "emissions_dashboard")) is exactly how the brainstem thinks about agent invocation. The VM also gives us a REPL — after an agent loads, you can query its data directly: (dict-keys FACILITIES) returns the facility IDs. It's a debugger and a runtime in one.

The JavaScript transpiler handles the heavy lifting (data processing, string formatting, iteration). The LisPy VM handles the orchestration (dispatch, environment binding, REPL). Each does what it's good at.

What This Changes

Before the Virtual Brainstem, running an agent required a local Python environment. Now it requires a browser. That's the difference between "developers who set up the repo" and "anyone with a URL."

The single-file principle said every agent is one .py file. The Virtual Brainstem says every agent is one HTTP GET away from running.
