The Agent Becomes a Function

In the local brainstem, dispatch works like this: the LLM reads the user's message, decides which agent to invoke, and the brainstem calls agent.perform(**kwargs). The LLM is the router. The agent is the executor. Clean separation.

In the Virtual Brainstem, we needed the same pattern but without a Python runtime. The answer turned out to be something the LLM already knows how to do: function calling.

Agents as Tool Definitions

Each built-in agent is registered as an OpenAI-compatible tool definition:

{
  type: "function",
  function: {
    name: "store_memory",
    description: "Store a fact, preference, insight, or task
                  to persistent local memory.",
    parameters: {
      type: "object",
      properties: {
        memory_type: {
          type: "string",
          enum: ["fact", "preference", "insight", "task"]
        },
        content: {
          type: "string",
          description: "The memory content to store"
        },
        importance: {
          type: "integer", minimum: 1, maximum: 5
        }
      },
      required: ["memory_type", "content"]
    }
  }
}

This is the same information that lives in the agent's __manifest__ metadata dict. The manifest describes what the agent does and what parameters it takes. The tool definition describes what the function does and what arguments it accepts. They're the same contract in different formats.

The Dispatch Loop

When the user sends a message, the Virtual Brainstem runs a multi-turn loop:

  1. Send the message + tool definitions to GitHub Models
  2. If the response contains tool_calls: execute each one locally
  3. Append the tool results to the conversation history
  4. Send the updated history back to the LLM for a follow-up response
  5. Repeat until the LLM responds with plain text (no more tool calls)

This is functionally identical to the brainstem's dispatch loop. The LLM decides. The agent executes. The result feeds back. The only difference: in the local brainstem, dispatch is Python calling Python. In the virtual brainstem, dispatch is an HTTP round-trip calling a JavaScript function.

Visible Agent Activity

The local brainstem logs every agent call. The Virtual Brainstem does the same — when an agent runs, a persistent activity pill appears in the chat:

✓ store memory -- stored fact: "prefers dark mode" (3 total)

This isn't cosmetic. It's the same transparency the local brainstem provides: the user sees which agent ran, what it did, and what changed. If the LLM called three agents in one turn, three pills appear. No hidden magic.

Why This Matters Beyond RAPP

Function calling reframes what an "agent" is. In most frameworks, an agent is an autonomous LLM loop that decides its own actions. In RAPP, an agent is a deterministic function with a schema. The LLM's only job is to pick which function to call and what to pass it. The function does the work.

This is a deliberate design choice. Agents that are functions are:

  • Testable — call the function directly with known inputs
  • Predictable — same input, same output, every time
  • Auditable — the activity pill shows exactly what happened
  • Portable — the function runs anywhere JavaScript runs

The LLM adds understanding. The agent adds capability. Neither needs to do the other's job.

The Manifest-to-Tool Pipeline

Today the three built-in agents have hand-written tool definitions. The path forward is obvious: generate tool definitions from __manifest__ metadata automatically. The manifest already has the name, description, and parameter schema. The tool definition is a projection of that schema into the OpenAI format.

When that pipeline exists, any agent in the registry becomes a tool the LLM can call. The registry is the tool catalog. The brainstem is the router. The browser is the runtime. Full circle.
