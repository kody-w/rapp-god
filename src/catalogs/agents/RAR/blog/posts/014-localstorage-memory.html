The Brainstem Remembers Without a Server

The local RAPP Brainstem stores memories in a file-based storage manager. The ManageMemory agent writes facts, preferences, insights, and tasks. The ContextMemory agent reads them back. It works because both agents share a filesystem.

The Virtual Brainstem has no filesystem. It has localStorage.

The Mapping

Local BrainstemVirtual Brainstem
storage_manager.write(key, value)localStorage.setItem('vb_memories', JSON.stringify(memories))
UUID-keyed JSON filesSingle JSON array under one key
Per-user via user_guidPer-browser (origin-scoped)
Survives rebootsSurvives tab closes, page refreshes, browser restarts
Shared across agents via storage factoryShared across agents via the same key

The mental model is identical: store a memory, recall it later. The persistence layer changed from disk to browser. The agent API didn't.

Memory Structure

Each memory is a JSON object:

{
  "id": "a7f2c...",
  "type": "preference",
  "content": "prefers dark mode and monospace fonts",
  "importance": 3,
  "tags": ["ui", "display"],
  "timestamp": "2026-04-04T21:10:00.000Z"
}

The store_memory agent appends to the array. The recall_memories agent reads it, optionally filtering by keywords, and sorts by recency. Both are plain JavaScript functions — no LisPy, no transpiler, just array operations on a JSON blob.

Why localStorage

We considered three alternatives:

  • IndexedDB — more storage, async API, but overkill for a flat list of memories
  • Server-side storage — requires a backend, breaks the zero-install promise
  • In-memory only — memories disappear on refresh, which defeats the purpose

localStorage gives us synchronous reads, 5-10MB of space (thousands of memories), origin-scoped isolation, and zero setup. It's the right tool for a browser-native agent that stores text.

The Privacy Angle

Memories never leave the browser. The LLM sees them only when the recall_memories tool is called — and only because the tool result is sent as a follow-up message. If you never ask "what do you know about me?", the memories stay in localStorage, untouched.

This is local-first memory. Your data lives in your browser, on your machine, under your control. Clear localStorage and it's gone. No account deletion form. No data export request. Just localStorage.removeItem('vb_memories').

What Carries Over

The local brainstem's memory agents have richer features: per-user GUID separation, tag-based retrieval, importance-weighted recall, session-scoped context. The Virtual Brainstem implements the same API surface but with browser-native persistence.

If you migrate from virtual to local, the conversation pattern is the same: "remember X" stores, "what do you know?" recalls. The agent names are the same. The tool schemas are the same. Only the storage backend changes — from a JSON array in the browser to a JSON file on disk.

That's the point. The brainstem doesn't care where memories live. It cares that it can store and recall them. Everything else is an implementation detail.
