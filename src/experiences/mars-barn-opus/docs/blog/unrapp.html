unRAPP: Why Your Agent Should Be One File That Runs Everywhere

kody-w · April 2025 · unRAPP Engineering

You have basic_agent.py. It runs on Python. Works great on your server.

Now you want it to run in a browser. In a virtual OS. On a Raspberry Pi. On a Mars rover simulator.

The traditional answer: rewrite it. Port the Python to JavaScript. Then port it again for the embedded runtime. Now you have three files that do the same thing. One gets updated. The other two drift. Six months later, they're three different agents wearing the same name.

unRAPP solves this: one file, any runtime, automatic translation.

The Problem: Digital Twin Drift

The AI-Agent-Templates library has 17 Python agents. Each is a class with __init__(name, metadata) and perform(**kwargs). They work great on Azure Functions.

But we built a LisPy VM that runs in browsers, in a virtual OS, on physical hardware, and inside a Mars colony simulator. We need those same 17 agents to run on the VM too.

The obvious approach: create 17 .lispy files alongside the 17 .py files. Now we have 34 files. When someone improves the Python calendar_agent.py, someone needs to also update calendar_agent.lispy. And vice versa.

This is the digital twin drift problem. Two files that should be identical but inevitably diverge because humans forget to update both.

The Solution: Don't Have Two Files

An agent is either a .py file or a .lispy file. You carry whichever one you prefer. When the other runtime needs it, unRAPP translates automatically. Behind the scenes. The user never sees it.

This works because both formats share the same interface contract:

Python (.py)
class CalendarAgent(BasicAgent):
    def __init__(self):
        self.name = 'calendar_agent'
        self.metadata = {
            "name": "calendar_agent",
            "description": "Mars calendar"
        }
    
    def perform(self, **kwargs):
        sol = kwargs.get('sol', 0)
        year = sol // 669
        sol_in_year = sol % 669
        season = (
            "Spring" if sol_in_year < 167
            else "Summer" if sol_in_year < 334
            else "Autumn" if sol_in_year < 501
            else "Winter"
        )
        return f"Year {year} Sol {sol_in_year} {season}"
LisPy (.lispy)
;; calendar_agent.lispy
(begin
  (define _agent_name "calendar_agent")
  (define _agent_desc "Mars calendar")
  
  (define year (floor (/ sol 669)))
  (define sol_in_year (% sol 669))
  (define season
    (cond
      ((< sol_in_year 167) "Spring")
      ((< sol_in_year 334) "Summer")
      ((< sol_in_year 501) "Autumn")
      (true "Winter")))
  
  (log (concat "Year " (string year)
    " Sol " (string sol_in_year)
    " " season)))

These are the same agent. Same name. Same logic. Same result. The Python version wraps it in a class. The LisPy version IS the logic directly. unRAPP converts between them because the mapping is mechanical:

Why LisPy Makes This Possible

LisPy is homoiconic — code and data are the same format. An S-expression is simultaneously:

  • A program you can execute
  • A data structure you can inspect, copy, and modify
  • A string you can embed in a JSON field
  • A text file you can read in any language

This means a Python program can contain a LisPy program as a string literal, and execute it via vm.run(). The Python class is just a wrapper. The LisPy is the soul.

# The Python wrapper IS the unRAPP translation:
class CalendarAgent(BasicAgent):
    PROGRAM = """(begin
      (define year (floor (/ sol 669)))
      (define season (cond ((< (% sol 669) 167) "Spring") ...))
      (log (concat "Year " (string year) " " season)))"""
    
    def perform(self, **kwargs):
        vm = LispyVM()
        for k, v in kwargs.items():
            vm.set_env(k, v)
        return vm.run(self.PROGRAM)

# The .lispy file IS the PROGRAM field extracted.
# The .py file IS the PROGRAM field wrapped.
# unRAPP moves between them. That's it.

Why Personal Preference Works

Some developers think in Python. They carry .py files. That's fine — the LisPy VM can import their perform() logic.

Some developers think in Lisp. They carry .lispy files. That's fine — a thin Python wrapper executes it on Azure.

Some developers start with LisPy in the browser OS, test it live, then need it on a server. unRAPP generates the .py.

Some developers have existing Python agents from AI-Agent-Templates. They want them in the browser. unRAPP generates the .lispy.

The user never thinks about translation. They think about their agent. The runtime handles the rest.

The Engineering Principle

Don't maintain digital twins. Maintain ONE source of truth and translate on demand. The translation is mechanical. The drift is eliminated. The user chooses their preferred format. The runtime adapts.

This principle applies beyond agents:

  • Configs: YAML ↔ JSON ↔ TOML — maintain one, convert for the consumer
  • APIs: REST ↔ GraphQL — maintain one schema, generate both interfaces
  • Docs: Markdown ↔ HTML — maintain one, render for the medium
  • Agents: .py ↔ .lispy — maintain one, unRAPP for the runtime

unRAPP is the specific case. The general principle is: don't duplicate what you can derive.

One file. One agent. Any runtime. unRAPP handles the rest.
