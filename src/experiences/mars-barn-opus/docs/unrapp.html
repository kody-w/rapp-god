unRAPP — The Universal Agent System

mars-barn-opus · 2025 · L100 L200 L300 L400

L100 What Is This?

unRAPP is a universal agent runtime that lets you write an agent once and run it everywhere — in a browser, from the command line, through Playwright automation, or on physical hardware. Every agent is a LisPy S-expression: a tiny program written in a safe, sandboxed Lisp dialect that the LisPy VM can execute in any environment. If you've used AI-Agent-Templates (the Python agent library), unRAPP is the next step: take those same agents, express them as LisPy programs, and run them anywhere without Python installed. The conversion is mechanical — same name, same interface, different runtime. You can maintain both formats from a single source of truth, or go all-in on .lispy and never touch Python. The system is invisible: it converts, routes, and executes behind the scenes so you can focus on what your agent does, not where it runs.

30-Second Example

Here is the simplest possible agent — basic_agent — in both its Python form (the AI-Agent-Templates original) and its LisPy form (the unRAPP equivalent). They do the same thing: accept input, log that they're ready, and return a status message. The LisPy version runs in the browser OS, the CLI, and Playwright with zero dependencies.

BASIC_AGENT.PY
from lib.basic_agent import BasicAgent
from lib.lispy_vm import LispyVM

LISPY_PROGRAM = """
(begin
  (define _agent_name "Basic Agent")
  (define _agent_desc
    "Simplest possible agent.")
  (define input
    (if (= _input "") "No input" _input))
  (log (concat "🤖 " _agent_name " activated"))
  (log (concat "📥 Input: " input))
  (log "✓ Agent ready")
  (concat _agent_name " processed: " input))
"""

class BasicAgentImpl(BasicAgent):
    def __init__(self):
        self.name = "Basic Agent"
        self.vm = LispyVM()

    def perform(self, **kwargs):
        self.vm.set_env("_input",
            kwargs.get("input", ""))
        return self.vm.run(LISPY_PROGRAM)

L200 How Do I Use It?

Path A: You Have .py Agents (AI-Agent-Templates Users)

If you've been using kody-w/AI-Agent-Templates, you already have Python agent classes that follow the BasicAgent pattern. unRAPP gives you a way to extract the decision logic into a portable .lispy file that runs anywhere — browser, CLI, Playwright, hardware — while keeping your .py wrapper intact as a compatibility layer.

Step 1: Identify Your Agent's Logic

Every AI-Agent-Templates agent has a perform(**kwargs) method. That method body is what becomes your .lispy program. The class scaffolding (__init__, self.name, self.metadata) maps to define statements at the top of the LisPy program.

Step 2: Run Your .lispy Agent

In the browser OS (os.html): Open the Terminal window. Paste the LisPy program directly and press Enter. The VM evaluates it immediately with whatever colony state is loaded.

;; Paste this into os.html → Terminal:
(begin
  (define _agent_name "Basic Agent")
  (log (concat "🤖 " _agent_name " activated on Sol " (string sol)))
  (concat "Ready on Sol " (string sol)))

Via Playwright SDK (automated testing / CI): Use window.os.exec() from a Playwright test to inject and execute a LisPy program in the browser OS context:

// playwright test — JavaScript
const { test, expect } = require('@playwright/test');

test('run basic agent', async ({ page }) => {
  await page.goto('docs/os.html');

  const result = await page.evaluate(() => {
    return window.os.exec(`
      (begin
        (define _agent_name "Basic Agent")
        (log "🤖 Agent activated")
        (concat _agent_name " ready"))
    `);
  });

  expect(result).toContain('Basic Agent ready');
});

Via CLI (Python): Use the LispyVM directly from Python. No browser needed:

# python3
from src.lispy import LispyVM

vm = LispyVM()
vm.set_env("sol", 42)
vm.set_env("o2_days", 15.0)
vm.set_env("_input", "Hello Mars")

with open("docs/agents/basic_agent.lispy") as f:
    program = f.read()

result = vm.run(program)
print(result)   # "Basic Agent processed: Hello Mars"
print(vm.output) # ["🤖 Basic Agent activated", "📥 Input: Hello Mars", "✓ Agent ready..."]

Path B: Starting Fresh With .lispy

If you're not coming from AI-Agent-Templates and want to start clean, LisPy is your only file. No Python required. No class hierarchy. No imports.

  1. Write your agent as an S-expression. The LISPY.md Rosetta Stone teaches the full language in one file. Every feature is documented with a working example. Drop it in your repo and any AI can read it too.
  2. Save it in docs/agents/. This makes it discoverable in the agent catalog. The index.json file lists all registered agents.
  3. Run it anywhere:
    • Browser OS: Open os.html, open Terminal, paste and run
    • CLI: python3 -c "from src.lispy import LispyVM; vm=LispyVM(); print(vm.run(open('docs/agents/your_agent.lispy').read()))"
    • Playwright: page.evaluate(() => window.os.exec(program))
    • In the simulation: The colony governor loads and runs .lispy programs every sol

Path C: You Want Both Formats

You maintain one file. The runtime handles the other. Here's how the relationship works:

  • The .lispy file IS the canonical logic — the perform() body, unwrapped.
  • The .py file is a thin wrapper that instantiates LispyVM, seeds environment variables from kwargs, and calls vm.run(LISPY_PROGRAM).
  • unRAPP is invisible. You edit the LisPy. The wrapper calls it. Both formats stay in sync because the Python wrapper is just a frame around the LisPy source.

Agent 2: context_memory_agent — Store & Recall State

Uses _mem_* environment variables to store, recall, list, and manage persistent memory across agent invocations. This agent demonstrates how LisPy programs maintain state through the VM environment — not through files or databases, but through named variables prefixed with _mem_.

PYTHON (.py)
from lib.basic_agent import BasicAgent
from lib.lispy_vm import LispyVM

LISPY_PROGRAM = """
(begin
  (define _agent_name
    "Context Memory Agent")
  (define _agent_desc
    "Stores and recalls context
     using VM env vars _mem_.")
  (define store_key
    (if (= _mem_action "store")
      _mem_key ""))
  (define store_val
    (if (= _mem_action "store")
      _mem_value ""))
  (cond
    ((= _mem_action "store")
      (begin
        (set! _mem_last_key store_key)
        (log (concat "💾 Stored: "
          store_key " = " store_val))
        (concat "Memory stored: "
          store_key)))
    ((= _mem_action "recall")
      (begin
        (log (concat "🔍 Recalling: "
          _mem_key))
        (log (concat "📦 Value: "
          _mem_value))
        (concat "Recalled " _mem_key
          ": " _mem_value)))
    ((= _mem_action "list")
      (begin
        (log "📋 Listing memories:")
        (log (concat "  Last stored: "
          _mem_last_key))
        "Memory list complete"))
    (true
      (begin
        (log (concat "🧠 "
          _agent_name " ready"))
        (log "Set _mem_action to:
          store, recall, or list")
        "Awaiting memory command"))))
"""

class ContextMemoryAgent(BasicAgent):
    def __init__(self):
        self.name = "Context Memory Agent"
        self.vm = LispyVM()

    def perform(self, **kwargs):
        self.vm.set_env("_mem_action",
            kwargs.get("action", ""))
        self.vm.set_env("_mem_key",
            kwargs.get("key", ""))
        self.vm.set_env("_mem_value",
            kwargs.get("value", ""))
        self.vm.set_env("_mem_last_key",
            kwargs.get("last_key", ""))
        return self.vm.run(LISPY_PROGRAM)

Agent 3: colony_governor_agent — The Core Decision Engine

This is the most important agent in the system. It reads colony state (oxygen, food, power, dust, risk), computes threat levels for each resource, normalizes them into allocation weights that sum to 1.0, and sets the four critical output variables: isru_alloc, greenhouse_alloc, heating_alloc, and food_ration. These allocations control how the colony divides its limited power budget — and the wrong allocation kills everyone.

PYTHON (.py)
from lib.basic_agent import BasicAgent
from lib.lispy_vm import LispyVM

LISPY_PROGRAM = """
(begin
  (define _agent_name
    "Colony Governor")
  (define _agent_desc
    "Adaptive resource allocation
     based on colony state, risk,
     and environmental conditions.")
  (log (concat "🏛️ Colony Governor
    — Sol " (string sol)))

  ;; Assess threat levels (0-1)
  (define o2_threat
    (if (< o2_days 15)
      (/ (- 15 (min o2_days 15)) 15)
      0))
  (define food_threat
    (if (< food_days 20)
      (/ (- 20 (min food_days 20)) 20)
      0))
  (define power_threat
    (if (< power_kwh 150)
      (/ (- 150 (min power_kwh 150))
         150) 0))
  (define heat_threat
    (if (> dust_tau 2.0)
      (/ (min dust_tau 5.0) 5.0)
      0.1))

  ;; Normalize
  (define total_threat
    (+ o2_threat food_threat
       power_threat heat_threat))
  (define safe (< total_threat 0.3))

  ;; Compute allocations
  (define isru_alloc_new
    (cond
      ((< o2_days 3)  0.85)
      ((< o2_days 7)  0.55)
      (safe           0.30)
      (true (+ 0.25
        (* o2_threat 0.4)))))

  (define greenhouse_alloc_new
    (cond
      ((< food_days 5)  0.60)
      ((< food_days 12) 0.45)
      (safe             0.40)
      (true (+ 0.20
        (* food_threat 0.35)))))

  (define heating_alloc_new
    (cond
      ((> dust_tau 4.0) 0.40)
      ((< power_kwh 50) 0.45)
      (safe             0.25)
      (true (+ 0.15
        (* heat_threat 0.25)))))

  ;; Normalize to sum to 1.0
  (define raw_total
    (+ isru_alloc_new
       greenhouse_alloc_new
       heating_alloc_new))
  (set! isru_alloc
    (/ isru_alloc_new raw_total))
  (set! greenhouse_alloc
    (/ greenhouse_alloc_new raw_total))
  (set! heating_alloc
    (/ heating_alloc_new raw_total))

  ;; Adaptive food ration
  (set! food_ration
    (cond
      ((< food_days 5)  0.5)
      ((< food_days 10) 0.7)
      ((< food_days 15) 0.85)
      (true             1.0)))

  ;; Return summary
  (concat "Governor Sol "
    (string sol) " — "
    "I=" (string
      (round (* isru_alloc 100)))
    " G=" (string
      (round (* greenhouse_alloc 100)))
    " H=" (string
      (round (* heating_alloc 100)))))
"""

class ColonyGovernor(BasicAgent):
    def __init__(self):
        self.name = "Colony Governor"
        self.vm = LispyVM()

    def perform(self, **kwargs):
        # Seed all colony state
        for k, v in kwargs.items():
            self.vm.set_env(k, v)
        result = self.vm.run(
            LISPY_PROGRAM)
        alloc = self.vm.get_allocation()
        return {
            "summary": result,
            "allocation": alloc
        }

L300 How Does It Work?

The Interface Contract

Both .py and .lispy agents share the same interface contract. Every agent has four things: a name, a description, parameters (environment variables it reads), and executable logic (the decision-making code). The formats express these identically:

ConceptPython (.py)LisPy (.lispy)
Agent nameself.name = "X"(define _agent_name "X")
Descriptionself.metadata["description"](define _agent_desc "...")
Read a parameterkwargs.get('key')Read env var key directly
Write an outputself.vm.set_env('key', val)(set! key val)
Return a resultreturn resultLast expression is the result
Log a messageprint(msg)(log msg)

They are equivalent because the Python wrapper's perform() body does one thing: call vm.run(LISPY_PROGRAM). The Python class is scaffolding. The LisPy program is the brain.

How the Python Wrapper Works

Every Python agent follows this exact pattern. There is no variation:

from lib.basic_agent import BasicAgent
from lib.lispy_vm import LispyVM

LISPY_PROGRAM = """
... the .lispy file contents, verbatim ...
"""

class MyAgent(BasicAgent):
    def __init__(self):
        self.name = "My Agent"
        self.metadata = {"description": "What it does"}
        self.vm = LispyVM()

    def perform(self, **kwargs):
        # 1. Seed environment variables from kwargs
        for key, value in kwargs.items():
            self.vm.set_env(key, value)

        # 2. Execute the LisPy program
        result = self.vm.run(LISPY_PROGRAM)

        # 3. Optionally read output variables
        allocation = self.vm.get_allocation()

        return result

That's the entire pattern. Every agent wraps a LisPy program. kwargs become environment variables. The program runs. The last expression is the return value. Output variables (like heating_alloc) are read back via get_allocation(). There is nothing else.

The LisPy VM

The LisPy VM (src/lispy.py) is a safe, sandboxed Lisp interpreter. It has these properties:

  • Homoiconic: Code is data. S-expressions are both executable programs and inspectable data structures. An AI can read a LisPy program, modify it, and generate new programs using the same syntax. The AST IS the source code.
  • Sandboxed: No file I/O. No imports. No network access. No eval of arbitrary Python. Pure computation only. A LisPy program cannot escape the VM.
  • Step-limited: Execution is capped at 10,000 steps. Infinite loops throw LispyError. No program can hang the system.
  • Cross-platform: The same VM runs in the browser (os.html, viewer.html), the CLI (python3 src/lispy.py), the simulation (src/sim.py), and physical hardware (Rappter). A program written anywhere runs everywhere.

The complete language reference is LISPY.md — the Rosetta Stone. Drop it in any repo and any AI that reads it immediately understands the full language. To see programs execute live, visit the Hall of Fame.

The colony splits its finite power budget three ways — more ISRU means less greenhouse means less food. Every allocation is a tradeoff. This is why the governor matters.

Writing Your Own Agent: Step-by-Step

Let's build a brand new agent from scratch: weather_report_agent. It will read the current dust, solar efficiency, and temperature data, determine a "weather forecast," and return a human-readable report.

Step 1: Choose a Name and Define the Contract

Every agent starts with a name and description. These are the first two define statements. The name becomes _agent_name and the description becomes _agent_desc:

;; weather_report_agent.lispy — Mars weather forecast from sensor data.
(begin
  (define _agent_name "Weather Report Agent")
  (define _agent_desc "Generates a Mars weather forecast from dust, solar, and temperature data.")
  ;; ... logic goes here ...
)

Step 2: Read Environment Variables

The VM environment is pre-seeded with colony state. Read the variables you need directly by name — no imports, no function calls:

  ;; Read sensor data from environment
  (define dust dust_tau)
  (define solar solar_eff)
  (define current_sol sol)

Step 3: Write the Decision Logic

Use cond for multi-branch decisions, if for binary choices, and math for computed values. Here we classify weather conditions and compute a visibility estimate:

  ;; Classify sky conditions
  (define sky_condition
    (cond
      ((> dust 4.0) "🌪️ DUST STORM — zero visibility")
      ((> dust 2.5) "🟠 HAZY — reduced visibility")
      ((> dust 1.0) "🟡 PARTLY CLEAR — moderate dust")
      (true         "🔵 CLEAR — excellent visibility")))

  ;; Estimate visibility in km (inverse of dust)
  (define visibility_km
    (if (> dust 0.1)
      (round (/ 20.0 dust))
      200))

  ;; Solar forecast
  (define solar_forecast
    (cond
      ((> solar 80) "☀️ Strong solar — full power expected")
      ((> solar 50) "⛅ Moderate solar — monitor panels")
      ((> solar 20) "☁️ Weak solar — power conservation advised")
      (true         "🌑 Minimal solar — activate reserves")))

Step 4: Format Output and Return

Use log for detailed output lines and return a summary string as the last expression:

  ;; Generate the report
  (log "🌤️ ═══ MARS WEATHER REPORT ═══")
  (log (concat "  Sol " (string current_sol) " Forecast"))
  (log "")
  (log (concat "  Sky:        " sky_condition))
  (log (concat "  Visibility: " (string visibility_km) " km"))
  (log (concat "  Dust τ:     " (string dust)))
  (log (concat "  Solar:      " solar_forecast))
  (log (concat "  Solar Eff:  " (string solar) "%"))
  (log "")

  ;; Advisory
  (if (> dust 3.0)
    (log "  ⚠️  ADVISORY: Limit EVA. Secure loose equipment.")
    (log "  ✅ No weather advisories."))

  (log "═══════════════════════════════")

  ;; Return summary (last expression)
  (concat "Weather Sol " (string current_sol) " — " sky_condition
    " — Vis " (string visibility_km) "km"))

Step 9: Write the Python Wrapper (Optional)

If you need your agent callable from Python (for CI, for integration tests, for the simulation loop), the wrapper writes itself:

from lib.basic_agent import BasicAgent
from lib.lispy_vm import LispyVM

class WeatherReportAgent(BasicAgent):
    def __init__(self):
        self.name = "Weather Report Agent"
        self.vm = LispyVM()

    def perform(self, **kwargs):
        for key, value in kwargs.items():
            self.vm.set_env(key, value)
        with open("docs/agents/weather_report_agent.lispy") as f:
            return self.vm.run(f.read())

Cartridge Export

An agent's entire state can be serialized as a cartridge — a portable snapshot containing three things:

  1. The program: the LisPy source code (the .lispy file)
  2. The environment: all seeded variables (colony state, input parameters)
  3. The memory: all _mem_* variables (persistent agent state)

A cartridge is a JSON object that looks like this:

{
  "program": "(begin (define _agent_name \"Colony Governor\") ...)",
  "env": {
    "sol": 42,
    "o2_days": 15.0,
    "food_days": 8.5,
    "power_kwh": 300.0,
    "dust_tau": 1.2,
    "colony_risk_index": 35.0
  },
  "memory": {
    "_mem_last_key": "crisis_count",
    "_mem_last_value": "3",
    "_mem_count": 7
  }
}

Import the cartridge into any VM instance to restore the agent exactly as it was — same program, same state, same memory. This enables agent sharing across colonies, migration between hardware and simulation, and replay for debugging.

# Export a cartridge
cartridge = {
    "program": open("docs/agents/colony_governor_agent.lispy").read(),
    "env": {k: v for k, v in vm.env.data.items()
            if isinstance(v, (int, float, str, bool))},
    "memory": {k: v for k, v in vm.env.data.items()
               if k.startswith("_mem_")}
}

# Import a cartridge
vm = LispyVM()
for key, val in cartridge["env"].items():
    vm.set_env(key, val)
for key, val in cartridge["memory"].items():
    vm.set_env(key, val)
result = vm.run(cartridge["program"])

Contributing to the Library

The agent library is community-grown. Here's how to add your agent:

  1. Fork mars-barn-opus
  2. Create your .lispy file in docs/agents/. Follow the naming convention: your_agent_name.lispy. Start with _agent_name and _agent_desc defines.
  3. Test it:
    • In the browser: open os.html, paste in Terminal
    • In Python: vm = LispyVM(); vm.set_env(...); vm.run(program)
    • Run the test suite: python3 -m pytest tests/ -q
  4. Add to index.json: append your filename to the "agents" array. If it's a port of an AI-Agent-Templates original, add an entry to "og_mapping". If it's Mars-native, add to "mars_native".
  5. PR back. Include a brief description of what your agent does and what environment variables it reads.
AGENT STYLE GUIDE

When writing agents for the library:

  • First line: ;; agent_name.lispy — One-line description. OG: original.py (or OG: Mars-native.)
  • First define: _agent_name
  • Second define: _agent_desc
  • Use log for human-readable output with emoji section headers
  • Use ═══ box-drawing for section separators
  • Return a one-line summary string as the last expression
  • Keep it under 80 lines if possible — an agent should do one thing well
