The 10 dismissals that come up first

Save yourself the conversation. These are the dismissals every skeptic produces in roughly this order. Short answer to each. Deep answers further down.

"Codex / Claude Code / Cursor already does this."
They write code. RAPP is the substrate the code lives in. Different layer; no overlap. Codex hands you 600 lines of Python; RAPP makes those 600 lines portable, sha256-pinned, sneakernet-shareable, multi-LLM-runnable.
"I'll just stitch the SDKs together."
SDKs are libraries that run inside one runtime. RAPP is a contract for what an agent IS that lets the agent's bytes travel between runtimes (and into Copilot Studio, sneakernet, airgapped sites). The most ambitious stitched stack solves 1 row of the table. See the SDK matrix below.
"MCP solves federation."
MCP is a protocol for an agent to call tools. RAPP is a contract for the agent itself — its identity (rappid), portability (single-file), distribution (sneakernet egg), governance (constitution + sha256). Layer above MCP, not the same as. RAPP agents use MCP. MCP doesn't give you any of the rest.
"Microsoft / OpenAI / Anthropic will build this eventually."
They have the opposite incentive. Their business model IS being the broker your agent talks through. A vendor will not ship a substrate that lets your agent leave their tenant — that's why this category gets built outside incumbents, not inside.
"If it's valuable, a big company would have done it."
New substrate categories are always built outside incumbents. Linux wasn't built by Microsoft. Git wasn't built by Atlassian. Kubernetes wasn't built by Amazon. The category exists precisely because no incumbent's incentive matched the substrate.
"Never heard of you. Tiny project."
Yes. That's the case for adoption now, not against it. The kernel is locked, the constitution is sealed, the contract is portable. The late-majority adopter gets the same contract a year-three adopter gets — they just pay for it in churn instead of curiosity.
"We use Slack / Notion / Confluence already."
Those are content stores with humans inside them. RAPP is computation that travels. Your Slack channel doesn't fork into a 70KB egg you hand to your customer's airgapped basement; your Notion doc doesn't sha256-verify into someone else's brainstem; your Confluence page doesn't run as a Copilot Studio agent.
"Show me a Fortune 500 customer."
There isn't one. There's Bill Whalen (Microsoft SE) testing it this week. By the time an F500 logo lands, the contract is sealed and you're competing for adoption against teams who started a year before you. Wedge buyers don't wait for case studies; case studies are the consequence of wedge buyers.
"Open source = no support / no vendor."
Bring your own vendor. RAPP is the substrate; your LLM provider (OpenAI / Anthropic / Azure / Copilot / local Ollama) is the engine. The substrate has no SaaS billing to maintain, no tenant that can be deplatformed, no support contract to renew. The only thing that can break is what you chose to use under it.
"We don't have a portability problem."
You will. Every time you switch IDEs, change LLM vendors, onboard a contractor, move to a new laptop, hand work to a coworker, open a customer engagement, or join a partner's project — the portability gap shows up. RAPP doesn't ask for portability; it makes it the default state.

Codex/Claude Code/Cursor make one developer faster at writing code. They were built for that. They are good at it.

What they don't do — what nobody else is doing — is give your team a portable, sneakernet-shareable, sha256-pinned, multi-LLM, federated workflow substrate that the AI's output can actually live inside. The agents your AI just wrote get stranded on your laptop. They don't compose. They don't travel. They don't share. They die when you switch tools.

The argument is: when Codex hands you the 600 lines it wrote for you, those lines need a substrate to live inside — something that makes them portable, sha256-verifiable, sneakernet-shareable, multi-LLM-runnable, federation-ready, and constitutionally-governed. That substrate doesn't come from your coding tool. It comes from RAPP.

"I'll just vibe-code my own."

Vibe-coding it yourself

What you have to build (and maintain forever)
  • Define an agent file format. Get it wrong twice. Settle on a third.
  • Build hot-discovery (or restart for every change).
  • Pick a vendor and lock in. Or build a provider-dispatch layer.
  • Build a distribution mechanism. (Pip? GitHub releases? URLs in Slack?)
  • Build a verification mechanism. (Sha256? Trust on first use?)
  • Build identity. (UUIDs? Random hashes? GitHub usernames?)
  • Build cross-tier portability. (Or pick one tier and live with it.)
  • Build offline / sneakernet path. (Or skip — "we have wifi.")
  • Build per-operator workspaces.
  • Build the trust web. (Or skip — "Slack works fine.")
  • Build governance. (Or skip — "we'll be careful.")
  • Build conformance tests for all of the above.
  • Maintain it. Forever. Across every team that adopts it.

Adopting RAPP

  • No public installer is currently shipped.
  • Drop your existing agent.py into ~/.brainstem/agents/.
  • Tell your brainstem what you want. It runs.
  • To share with a teammate: chat "pack the workspace as an egg." Send them two files.
  • To deploy to Copilot Studio: same .py. Tier 3. Unmodified.
  • Everything above (rappid, federation, sha256, conformance, two-tier estate, multi-LLM, sneakernet, governance) is already built. You inherit it.

"But the SDKs already have all this stuff."

No. The SDKs give you frameworks for writing agents. Each is opinionated about its own runtime, its own state model, its own vendor relationship. None of them give you a contract for what an agent IS that's portable across all of them — and that's the gap.

Why "stitch the SDKs" doesn't actually work

Suppose you take the most ambitious version: LiteLLM + LangGraph + your-own-Modal-sandbox + Auth0 + a HuggingFace mirror + Pydantic for types. Six SDKs glued together, each from a different vendor, each with its own deprecation cycle.

You now have:

  • Multi-LLM dispatch ✓ (LiteLLM)
  • One row solved.

You still don't have:

  • ✗ A single-file agent format that hot-loads with no restart
  • ✗ The same agent code running in T1 (laptop) + T2 (Azure) + T3 (Copilot Studio) unmodified
  • ✗ A sneakernet cartridge format (.egg) that drops onto a peer's machine in two files
  • ✗ Sha256-pinned manifest distribution (no SDK has this; they ship Python packages)
  • ✗ A persistent organism identity that survives kernel upgrades (rappid)
  • ✗ Pure raw-fetch federation (no SDK lets a stranger discover your agents via curl + no API)
  • ✗ A two-tier mandatory estate boundary for PII (no SDK has this concept)
  • ✗ Bilateral correspondence channels with no SaaS broker
  • ✗ A constitution + conformance gates the runtime enforces

The SDKs are libraries. RAPP is a contract — a constitutional definition of what an agent IS, that makes the agent's bytes portable across every one of those SDKs and across substrates the SDKs can't reach (Copilot Studio, sneakernet, LAN-only mesh, airgapped customer sites). LiteLLM is the only one of them that's the same shape as a RAPP primitive — and RAPP just uses LiteLLM-style dispatch as one row of its substrate.

The category mistake: comparing RAPP to an SDK. They're different layers. An SDK helps you write an agent. RAPP defines what the agent IS so it can travel.
