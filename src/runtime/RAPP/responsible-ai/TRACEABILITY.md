# RAPP — RAI-to-Standard Traceability Matrix

> **Owner:** \<to be assigned\> · **Status:** DRAFT · **Last updated:** 2026-06-27 · **Maps to:** A1, A2, A3, A4, A5, T1, T2, T3, F1, F2, F3, RS1, RS2, RS3, PS1, PS2, I1 + Microsoft Generative-AI supplement

This is the at-a-glance compliance board for RAPP (the Rapid Agent Prototype Platform) against the **Microsoft Responsible AI Standard v2** (June 2022) — six principles, 17 Goals — plus the Microsoft **Generative-AI supplement**. Every other artifact in this directory hangs off a row here.

## Honest posture

**RAPP does not pass the Microsoft Responsible AI Standard today.** This matrix exists to make that legible, not to assert compliance. Read every row as a trajectory: **current state → what the Standard requires → exit criteria**. P0 rows are absolute blockers to any RAI sign-off; none of them is closed.

Two grounding caveats are preserved throughout, deliberately, rather than papered over:

1. **The Generative-AI requirements are not a numbered public clause list.** Microsoft's GenAI supplement is mapped here *by intent* (RAI impact assessment for GenAI, content/abuse filtering, Prompt Shields, metaprompt & grounding, pre-release red-teaming, ongoing abuse monitoring, AI-content disclosure, staged release) and to the publicly documented Azure AI Foundry operationalization — **not** to fabricated clause IDs.
2. **Standard sub-clauses are referenced by role, not quoted verbatim.** The official Standard PDF did not extract cleanly in the source analysis, so sub-clauses such as RS1.4 (operational ranges), RS2.1 (predictable-failure catalog), and A5.1 (oversight) are referenced by their *role*, not reproduced word-for-word.

One honest counterweight also runs through the matrix: Tier-1 inference flows through the GitHub Copilot API, which applies Microsoft/GitHub's **own** upstream content filtering and abuse monitoring on the completion — so RAPP is not at literal zero protection. But that layer is upstream, unconfigurable, unevidenced, blind to agent-injected system prompts, and does **not** gate local tool execution, so it cannot satisfy the Standard on RAPP's behalf.

**Gap-severity legend:** **P0** = blocks RAI sign-off · **P1** = hardening the contained system depends on · **P2** = scale / monitoring / inclusiveness · **P3** = polish & maintenance cadence. Phases (P0–P3) below correspond to [ROADMAP.md](./ROADMAP.md).

## How to read this board against the roadmap

- **P0 rows are the only true blockers.** They cluster into: the three governance artifacts (A1 Impact Assessment, A2 Sensitive Uses, T2 Transparency Note), the four compounding runtime-security findings (PS2 sandbox, auto-install, Cave provenance, network/auth), the safety + injection layer at `/chat` (RS1 + GenAI), and the two privacy/disclosure blockers (PS1 data-flow disclosure, PS1 `/diagnostics` exfil). The four runtime-security items are P0 because they **compound** — the Cave delivers untrusted code, auto-install executes it, no sandbox contains it, no auth stops a remote LAN trigger; the chain is the blocker, not any single link.
- **Everything in P1–P3 is hardening, scale, and polish** that the P0 artifacts will themselves schedule.
- **Single chokepoints to build on:** the one `/chat` seam (one place to retrofit safety + disclosure), the lifecycle `confirm:true` handshake (the exact pattern to extend to `run_tool_calls` for A5), sha256 pins + KERNEL_AGENTS protection (a hook for real signing), and the rappid lineage key (a provenance/signing key already in the ecosystem).
